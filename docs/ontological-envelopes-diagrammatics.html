Ontological Envelopes — Directory Systems Diagrammatics

Generic structural comparison of three directory governance architectures. Non-branded.

I. The Generic Envelope Model

All three systems share a common five-layer architecture. They differ in how each layer is instantiated, governed, and enforced.

Universal Layer Architecture
Layer 0
Sealed
Substrate
Governor: Vendor exclusively  |  Membrane: Cryptographic seal or integrity protection
Mutability: None (or admin-only with integrity checks)  |  Durability: Replaced atomically by vendor updates
Principle: The system-as-distributed is the ground truth. No drift permitted between intention and state.

Extension
Layer

II. Three Instantiations

Same model, three governance philosophies. Each column shows how one system fills the generic layers.

Comparative Instantiation
Editorial System
The Curated Museum
Sealed
Crypto-sealed snapshot. Vendor-signed, hash-verified at every block read. No process can write — including root.
Extension
Admin-gated directories for machine-wide apps and libraries. Package manager uses a vendor-specific alternative path on newer hardware.
Visible
8 provisioned directories: Presentation, Artifacts, Ingress, 3 Material Buckets, Sharing Surface, Hidden Mechanism. Vendor assigns magic icons.
Mechanism
Hidden since 2011. Three sub-envelopes: Disposition (API-managed, never hand-edit), Capacity (per-app data), Accident (expendable, system may purge).
Ungoverned
Dotfile stratum. Perceptual membrane hides all dot-prefixed entries. System pretends this layer does not exist. Zero governance.
Truth
Proprietary APIs. Filesystem is storage only. System state accessed through framework calls, not file reads.
Mediators
Perceptual Lens (curates view) · Access Gate (per-app per-dir) · Locality Dissolver (cloud sync of Artifacts + Presentation)
Bureaucratic System
The Registry Kingdom
Sealed
ACL-guarded, not sealed. Integrity protection via resource protection service. Mutable in principle — guarded, not immutable.
Extension
Two paths: vendor-namespaced program dirs (per-architecture) and machine-wide data dir. Admin + elevation prompt required.
Visible
12+ provisioned directories (7 active, 5+ legacy). Abstracted via GUID system — paths are indirected, never hardcoded. Cloud service can absorb Artifacts + Presentation.
Mechanism
Three-tier model classified by portability: Roaming (follows user across machines), Local (machine-specific), Low (sandboxed process). Most precise classification of the three systems.
Ungoverned
Dotfile stratum. No enforcement. No convention. Unix-origin tools dump dotfiles in profile root. System has no opinion.
Truth
Hierarchical binary database. Hardware, services, config, app registration all in one opaque store. Filesystem and database can disagree.
Mediators
Perceptual Lens (curates + localizes) · Elevation Gate (per-action) · Locality Dissolver (folder redirection / cloud KFM)
Transparent System
The Self-Describing Machine
Sealed
Package-managed, optionally immutable. Traditionally mutable; immutable-root variants seal it as versioned artifact. Content-addressed variants make paths intrinsic.
Extension
Two models: integrated hierarchy (admin-compiled, mirrors system structure) vs self-contained bundles (vendor-owned subtrees). Package manager governs both.
Visible
NO provisioned directories. Home dir is empty. User creates own structure. Specification exists (voluntary) but system enforces nothing.
Mechanism
Voluntary specification separating config, data, state, cache, runtime. Precise semantics but zero enforcement. Compliance is cultural, not technical.
Ungoverned
Dotfile stratum. Largest of the three — includes all non-spec-compliant tools. Home directory is a contested zone between spec and legacy.
Truth
The filesystem itself. Processes = files. Hardware = files. Config = files. Being and description are unified. Filesystem IS the system API.
Mediators
Perceptual Lens (optional, desktop-dependent) · POSIX permissions (coarse) · Kernel namespaces (true isolation — unique to this system)

IX. The Mold-Breakers — Systems That Reject the Model

Radical Alternatives
TRADITIONAL MODEL: Location is identity. /usr/bin/python = "the system's Python." Path encodes role. Hierarchy is taxonomy. ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── MOLD-BREAKER 1: CONTENT-ADDRESSED STORE /store/<HASH>-python-3.11.7/bin/python Path encodes PROVENANCE, not role. Hash = f(source, dependencies, build flags). Same inputs → same hash → same path on any machine. Identity is INTRINSIC (content-determined), not POSITIONAL (path-determined). Multiple versions coexist. "Install" = symlink from profile to store path. Updates are atomic. Rollbacks are instant. Garbage collection reclaims unused paths. Breaks: FHS positional hierarchy. "Where you are" no longer means "what you are." Extends to $HOME via declarative home manager → home directory becomes a derivation. MOLD-BREAKER 2: KERNEL NAMESPACES (filesystem virtualization) ┌──────────────────────┐ ┌──────────────────────┐ │ NAMESPACE A │ │ NAMESPACE B │ │ / │ │ / │ │ ├── bin/ │ │ ├── bin/ │ │ ├── etc/ │ │ ├── etc/ │ ← Different /etc! │ ├── home/user/ │ │ ├── app/data/ │ │ └── ... │ │ └── ... │ └──────────┬───────────┘ └──────────┬───────────┘ │ │ └────────── same kernel ──────┘ Two processes on one machine see DIFFERENT FILESYSTEMS. Not different directories — different REALITIES. Overlay filesystem layers compose read-only base + writable top. True isolation: one namespace cannot perceive another. Breaks: The assumption that there is ONE hierarchy. Multiplies the envelope model. Only available on the Transparent system. Editorial and Bureaucratic lack this primitive. MOLD-BREAKER 3: PORTABLE ENCRYPTED IDENTITY ┌─────────────────────────────────────────────┐ │ ENCRYPTED HOME DIRECTORY IMAGE │ │ │ │ User record (UID, GID, password hash) │ │ Configuration │ │ Data │ │ All state │ │ │ │ The home directory IS the identity. │ │ Carry it on USB. Plug into any machine. │ │ You exist. │ └─────────────────────────────────────────────┘ │ ▼ System provides runtime (kernel, services). Identity provides the person (home directory, credentials). The user OWNS THEMSELVES — system merely hosts. Breaks: The assumption that the system owns the user (/etc/passwd defines identity). Inverts the relationship: identity is portable, system is interchangeable. ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── ── WHAT THEY SHARE: All three mold-breakers attack the same assumption: that the directory hierarchy is FIXED, SINGULAR, and SYSTEM-OWNED. Content-addressed: paths are computed, not assigned. Namespaces: hierarchies are multiplied, not shared. Portable identity: home directories are owned, not provisioned.
