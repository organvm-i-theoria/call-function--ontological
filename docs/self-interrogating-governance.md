# Self-Interrogating Governance

**Source:** Session 2026-04-04 — evolved through three iterations:
1. Definitive governance manifest (if/or/else + 12 contradictions) → rejected as "too decided"
2. Self-interrogating question set → accepted as the correct form
3. Living materia concept → the abstract ground
4. Functional implementation (sprout→proven) → the concrete realization

---

## Principle

The system asks itself questions. The answers produce rules. The rules last until the next answer. Nothing below is governance. Everything below is a question the system asks itself. The answer — demonstrated, not asserted — is the governance for this iteration.

---

## On What Exists

- What is here? → inventory. not classification. not judgment. what is here.
- What does each thing do? → stated by the thing itself, or observed. not assigned.
- What would be different if this thing were removed? → if nothing: the thing may not exist in the way it claims. if something: that something is the thing's function.
- Does this thing know it is here? → if yes: how? what artifact says so? if no: is it a thing, or is it debris?

## On Boundaries

- Where does this end? → point to the boundary. if you cannot point, there is no boundary.
- What crosses inward that this did not request? → whatever crosses unrequested defines leakage.
- What crosses outward that nothing receives? → whatever leaves unreceived defines waste.
- Is this boundary drawn or discovered? → if drawn: by whom? is the drawing justified by observation? if discovered: what observation revealed it? if neither: the boundary is a guess.

## On Coupling

- What interacts with what? → show the data that crosses. not the schema. the data.
- Are there interactions not recorded anywhere? → how would you detect one? if you cannot detect unrecorded interactions, your record is unfalsifiable.
- If this coupling were severed, what changes? → if nothing: nominal coupling. if everything: load-bearing coupling. the ratio is a measurement.

## On Containment

- Why is this inside this and not something else? → if "because I put it there": thesis. if "because it coupled more tightly here": emergence. if no answer: the containment is arbitrary.
- Does the container know what it contains? → if yes: show the knowledge artifact. if no: it is a directory, not a container.
- Does the contained know what contains it? → if yes: it depends on its container. is that dependency declared? if no: it is portable. is that portability real?
- What does the container gain from this member that it could not gain from a different member? → if nothing: interchangeable. if something: that something is the member's contribution.

## On Identity

- What is the smallest change that makes this a different thing?
- What is the largest change that keeps this the same thing?
- The zone between these two answers is the identity. The zone is a measurement, not a definition.
- If you deleted the description and rebuilt it from observation, what would be different? → the difference is the gap between claim and reality. the gap is data.
- Does identity come from config, function, coupling, or position? → show evidence for each. do not pick one. the weight shifts across iterations. that shift is the measurement.

## On Proof

- Can this demonstrate what it claims? → if yes: demonstrate. the demonstration is the proof. if no: what prevents demonstration? the obstacle is either a limit of the system or a limit of the claim. which one?
- Is the demonstration logical, mathematical, computational, or empirical? → it may be all four. it may be one. the system does not prescribe which. the thing itself determines what kind of proof it can carry.
- What can this NOT prove about itself? → the answer is the completeness boundary. an empty answer is the most dangerous answer. nothing that knows itself believes it knows everything.
- Has this proof changed since last iteration? → if yes: what changed? the proof or the thing? if no: is the thing static, or is the proof stale?

## On Change

- What is different from last time? → the delta. not "what was decided." what IS different.
- Which changes were caused by the user? Which by the system? Which by neither? → "neither" is the most interesting answer. it means something changed that no agent intended.
- If you reverted to three iterations ago, what would be inconsistent with the current neighbors? → the inconsistencies reveal coupling. coupling that only shows under reversion is coupling that was never declared.
- What parameter, moved to the opposite extreme, would teach the most? → the answer names the next experiment.

## On Necessity

- Why does this exist instead of not existing? → "because I created it" is not an answer. "because this need exists that nothing else serves" is. "I don't know" is valid, accompanied by a method to find out.
- Is this here because it was needed, observed, or expected? → needed: show the need. observed: show the observation. expected: acknowledge the thesis. test it. if you cannot distinguish: that inability is data.
- What is the cost of this thing's existence? → every thing imposes coupling load, proof obligation, naming weight, attention cost. if the cost exceeds the function: the thing is debt.

## On the Whole

- Does traversal of the hierarchy yield complete knowledge? → test it. traverse. what did you learn? what did you NOT learn that you expected to? the gap between expectation and learning is the completeness boundary.
- Does this system contain the tools that govern it? → if yes: the system is self-referential. acknowledge the loop. do not resolve it. if no: what governs it? is the governor visible?
- Is the hierarchy a description or a prescription? → if description: does it match what's here? if prescription: does what's here match it? the answer may be different for different parts of the same hierarchy. that's data.
- What would a stranger see? → not what the architect intends. what someone with NO context would observe. the difference between intent and observation is the system's opacity.

## On the Ideal

- How does this instance differ from the others? → the differences are not errors. they are dimensions.
- What do all instances share? → the invariant is the skeleton.
- What do all instances differ in? → the variance is the life.
- Is the ideal a destination or a difference? → if destination: what evidence points to it? if difference: what evidence reveals the shape of the space? the answer changes. the question does not.

## How the System Uses This

Ask all questions. Record all answers as artifacts (not assertions). Compare answers to previous iteration's answers.

The delta between answers = measurement. The pattern of deltas across questions = structure. The structure across iterations = the governance that emerged.

Nothing in this manifest is a rule. Everything in this manifest is a question. The answers are the rules. The answers change.

## The 12 Contradictions (From the Definitive Draft)

Before this self-interrogating form was reached, a definitive if/or/else governance manifest was produced and rejected as "too decided." That manifest identified 12 structural contradictions between the brainstorm's own rules. These contradictions remain valid as design tensions — they are questions the system must continuously answer, not problems to solve once.

### 1. Self-Description vs. Incompleteness
The hierarchy claims traversal yields complete knowledge (Axiom 6). The proof system requires acknowledging incompleteness (completeness-boundary). Resolution: completeness is bounded, not absolute.

### 2. Every Directory Is a System vs. Emergence
Axiom 1 implies system.yaml at creation. Emergence says full schema comes last. Resolution: the schema grows with the system — minimal seed at creation, full schema at formalization.

### 3. Pre-declared Domains vs. Earned Domains
Architecture pre-declares 6 domains. Emergence forbids pre-declaration. Resolution: the 6 domains are a prediction to test against emergence. Convergence validates; divergence reveals better categories.

### 4. Pre-declared Levels vs. Emergent Hierarchy
Same as #3 on the depth axis. Resolution: levels are observations about recursion depth, not inputs.

### 5. Level-Prefixed Naming vs. Emergence
Names cannot carry level prefixes before levels are earned. Resolution: prefixes are annotations added retroactively, never mandatory.

### 6. Undeclared Coupling = Defect vs. Observed Coupling
Observation window violates Axiom 5. Resolution: introduce coupling status (pending → declared → deprecated). Axiom 5 applies to stable state, not observation window.

### 7. Full Proof Obligation vs. Minimal Seed
4-field seed cannot carry 5-artifact proof directory. Resolution: proof obligation grows with maturity. EXISTS check at creation, full apparatus at formalization.

### 8. Self-Containing Loop vs. Substrate Limitation
Container requires self-reference. Container cannot fully describe what it depends on. Resolution: acknowledge the loop via completeness-boundary. The incompleteness IS the integrity.

### 9. Convergence vs. Ideal-as-Difference
Individual convergence (each instance refines itself) vs. ensemble divergence (instances explore phase space). Resolution: convergence is local (within instance), divergence is global (across instances). The ideal is the manifold shape, not any point on it.

### 10. Containment = Understanding vs. Containment = Reification
Containment simultaneously achieves and threatens understanding. Resolution: the distinction is in the architect's stance, not in the container. Container must include declaration that it is a model.

### 11. Social Domain vs. Social Systems Resist Containment
Architecture includes social domain. Philosophy says social systems resist containment. Resolution: social domain is included but flagged as requiring different epistemological treatment — provisional limits, empirical proof, higher revision frequency.

### 12. Emergence Ladder = Value Hierarchy vs. Emergence-First
Stacking domains implies prescriptive value claim. Emergence says don't stack until observed. Resolution: the ladder is a conjecture. Record predicted order vs. observed order. Delta = data about architect's assumptions.

## The Living Materia — Abstract Ground

The container does not hold things. The container holds the capacity to become things.

The materia is undifferentiated substance that takes form under two forces: **need** (the system requires something — a gap in the coupling graph) and **want** (the architect desires something — no structural justification, only vision). Need is structural and detectable. Want is intentional and introduced only by the human.

The unfolding is autocatalytic — each transformation creates conditions for the next. The container does not grow by addition. It grows by generation. The arrow does not reverse. Nothing is deleted. Everything is composted — substance remains, form dissolves.

Three kinds of shapes emerge:
- **Needed** — fills a structural gap. Functional. Detectable.
- **Wanted** — fulfills the architect's vision. Intentional. Only human-introduced.
- **Emerged** — arose from interaction of other shapes. Neither called nor anticipated. The surplus is the life.

The materia is attention made durable. Every act of naming, shaping, coupling crystallizes attention into form. The collection of forms is a fossil record of everywhere attention has been. The materia replicates because attention generates more attention.

## Functional Implementation — How the Materia Becomes Buildable

The materia is continuous. The filesystem is discrete. Every act of implementation is a cut. The question: how do you cut without killing?

### Three Simultaneous Requirements

1. **Addressable** — you must be able to point at a region and say "this." Requires boundaries.
2. **Verifiable** — each region must demonstrate what it does. Requires stable identity.
3. **Fluid** — regions must transform, replicate, merge, decompose. Requires soft boundaries.

Boundaries must be hard enough to address and verify, soft enough to move.

### The Manifest — The Only Required Artifact

One file per region. Four fields at creation:

```
what: (what this region does — plain language, one line)
takes: (what it needs from outside)
gives: (what it produces for outside)
alive: (yes/no — is this region actively transforming?)
```

### Growth Stages (Capabilities, Not Pipeline)

**Sprout** — the four fields. Exists. Does something. That is all that is known. Verified by existing.

**Rooted** — couplings become visible. Manifest gains `couples_with`. Couplings added when observed, not planned. Verified by showing data crosses declared couplings.

**Branching** — internal structure develops. Subregions appear. Manifest gains `contains`. Parent knows children exist, not their internals. Verified by children existing with their own manifests.

**Fruiting** — outputs that others depend on. Manifest gains `interface: { stable, experimental }`. Stable outputs will not change without notice. Others may depend on stable, not experimental. Verified by stable outputs matching declaration.

**Proven** — can demonstrate its claims. Manifest gains `proof: { demonstrates, method, last_verified, boundary }`. Proof is self-defining — the region decides what it can prove. Verified by the demonstration holding. The `boundary` field declares what CANNOT be proven — the completeness limit.

Stages are not linear. A region may fruit before branching. May prove something before couplings recorded. May decompose back to sprout.

### Verification Scales With Maturity

- Sprout: does it exist? does the manifest have 4 fields?
- Rooted: does data cross declared couplings?
- Branching: do declared subregions exist with their own manifests?
- Fruiting: do stable interface outputs match declaration?
- Proven: does the declared proof hold?

### Three Maintenance Operations

**Composting** — region loses all couplings, `alive: no`. Not deleted. Marked as available for decomposition. Substance remains for new growth.

**Grafting** — region couples more tightly with different parent. Move directory, update both manifests, record the graft.

**Splitting** — region too large to verify as one. Interior differentiates into subregions with own manifests. Parent becomes container.

### The Single Invariant

The only mandatory maintenance across all maturity levels: **the manifest matches reality.** Not the other way around. One law: do not lie about what is here.

### Provability

- **Locality** — every proof is local to its region. Failure is local.
- **Proportionality** — proof obligation scales with claims. A region that claims nothing proves nothing.
- **Composability** — parent's proof = conjunction of children's proofs + composition proof. Same structure as mathematical induction.

The system is never complete. The system is always functional. A forest is never complete. A forest is always functional.
